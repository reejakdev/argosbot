Every incoming message flows through a strict privacy-preserving pipeline before any LLM sees it. The pipeline is fully transparent and auditable.

Key insight
The cloud LLM (Claude, GPT-4o…) only ever sees anonymized placeholders, never raw content. Your partner's name appears as [PERSON_1], their address as [ADDRESS_1]. The lookup table stays on your machine.
// privacy model

Zero-egress by design

Privacy isn't a setting you enable — it's the default architecture. Raw message content never leaves your machine.

🔴
Raw messages → never stored
Incoming messages are processed in memory and discarded. Only a SHA-256 hash and metadata are persisted in the database — never the content itself.
🟡
PII → anonymized before any LLM
A two-pass anonymizer runs before anything else: regex for structured patterns (addresses, amounts, emails), then an optional local LLM for names, companies, and project references. The lookup table is in-memory only.
🟢
Cloud LLM → receives only placeholders
Claude or GPT-4o only ever sees anonymized text. It reasons over [PERSON_1], [AMOUNT_1] — not the actual values. De-anonymization happens locally after the response.
🔵
Approvals → you execute, Argos proposes
The planner drafts actions — it never fires them. A proposal sits in the approval queue until you explicitly approve it from the web dashboard or Telegram bot.

// multi-model

Route by role, not by convenience

Different pipeline stages need different models. Argos lets you assign each role — sanitize, anonymize, classify, plan — to a different provider. Mix local and cloud to get the best of both.

A local 7B model (Ollama) is fast enough for sanitization, classification, and anonymization — tasks that need to see raw content. Keep the capable cloud model (Claude, GPT-4o) for planning and reasoning, where it only sees already-anonymized text.

Sending raw content to a cloud LLM to "anonymize" it defeats the entire privacy model — Argos enforces that the anonymizer never runs on a cloud model.

// privacy config — route by role
"privacy": {
  "provider": "ollama", // local model
  "model": "qwen2.5:7b",
  "roles": {
    "sanitize": "privacy", // local
    "llmAnon": "privacy", // local
    "classify": "privacy", // local
    "plan": "primary" // cloud, anon only
  }
}

// features

Built for operators

Argos is designed for people who deal with high message volume across multiple channels and need an assistant that acts as a second brain — without ever going rogue.

🎯
Triage engine
Every message is classified: task, reply needed, transaction whitelist request, or ignore. Urgent items are flagged and surfaced first via your approval channel.
🧩
RAG-backed drafts
When a reply is needed, Argos searches your personal memory and knowledge base first. Drafts are grounded in context — not generic LLM output.
📚
Knowledge base
Index URLs, GitHub repos, Notion pages, or local files. Stored as vector embeddings via Ollama. Argos searches this context before drafting any response.
🧠
Personal memory
Argos remembers facts about your contacts and ongoing threads. Memory is stored locally and searched semantically — never sent to a cloud service in raw form.
📋
Proposal queue
Every planned action — reply, calendar event, Notion page, email — becomes a proposal with full context. Review, edit, approve or reject from the dashboard or Telegram.
🔌
Plugin system
Write custom plugins in TypeScript. Receive message hooks, emit proposals, add custom tools to the planner. The registry is hot-pluggable at runtime.
🛠
MCP integrations
Connect Notion, Google Calendar, Gmail, Outlook, Supabase, Linear, browser automation, 1Password, Slack and more via the Model Context Protocol catalog.
💬
Briefing mode
Daily or on-demand briefings summarizing your open tasks, pending proposals, and priority messages — delivered to your Telegram bot or the web app.
🔐
WebAuthn dashboard
The approval web app runs locally over TLS. Authentication uses WebAuthn (passkey/TOTP) — no passwords, no exposed ports required.

// getting started

Up in minutes

Argos is a Node.js process that runs on your machine. No cloud service to sign up for, no data leaves your infrastructure.

STEP 01
Clone and install
Requires Node.js 22+. All dependencies install locally.
git clone https://github.com/reejakdev/argosbot
cd argosbot
npm install
STEP 02
Run the setup wizard
Interactive CLI guides you through channels, LLM provider, and integrations. Creates ~/.argos/config.json.
npm run setup
STEP 03
Start Argos
Argos connects to your channels, starts monitoring, and opens the approval dashboard at https://localhost:3000.
npm run dev  # dev with hot reload
npm start    # production
OPTIONAL — LOCAL PRIVACY MODEL
For maximum privacy, run a local model via Ollama to handle sanitization and anonymization. The cloud model will only ever see anonymized placeholders.
# Install Ollama from ollama.ai, then:
ollama pull qwen2.5:7b
ollama pull nomic-embed-text
ollama serve

// why argos

Why not just use a regular AI assistant?

Most AI assistants give the LLM full access to your data and act on your behalf in the background. Argos is built on the opposite principle.

Capability Argos Generic AI assistant
Raw messages reach the cloud LLM ✗ Never ✓ Always
PII anonymized before LLM call ✓ Two-pass (regex + local LLM) ✗ Not by default
Actions fire without your approval ✗ Never ⚠ Often by design
Multi-channel (Telegram, email, WhatsApp, Slack, Discord) ✓ ⚠ Rarely all at once
Runs entirely on your machine ✓ No cloud service to sign up for ✗ Hosted SaaS
Mix local + cloud LLMs per pipeline stage ✓ Per-role routing ✗ Single provider
Immutable audit log of every event ✓ SQLite append-only ✗ Usually none
Extensible with custom plugins ✓ TypeScript plugin API ⚠ Depends on platform
Open source ✓ MIT ✗ Usually closed
The problem

Standard AI assistants that read your messages send your raw conversations — names, addresses, amounts, business context — to cloud APIs. You have no control over what gets stored or how it's used.

The Argos answer

Argos interposes an anonymization layer between your raw messages and any cloud LLM. Claude reasons over [PERSON_1] and [AMOUNT_1] — never the actual values.

The constraint

This isn't a magic privacy shield — it's a deliberate architectural choice with tradeoffs. The LLM sees less context, which means slightly less precise outputs. That's the cost of not leaking your data.

Frequently asked questions

Raw message content never leaves your machine. What gets sent to a cloud LLM is only the anonymized version — with all names, addresses, amounts, and identifiers replaced by placeholders like [PERSON_1], [ADDRESS_1], [AMOUNT_1].

The lookup table that maps placeholders back to real values is held in memory only — it's never persisted to disk or sent anywhere. After the pipeline processes a message, the raw content is discarded. The database stores only a SHA-256 hash of the original message and its anonymized summary.

If you configure a local model (Ollama) for the privacy roles, then even the anonymized text never leaves your machine — the full pipeline runs locally.

No. Ollama is entirely optional. Argos works with just a cloud API key (Anthropic, OpenAI, etc.).

Without a local model, the first anonymization pass is regex-only — structured patterns (blockchain addresses, emails, phone numbers, amounts) get redacted, but implicit references like names or project codes may not. The cloud LLM still never sees raw content, just the regex-anonymized version.

With a local model (recommended), a second LLM pass runs on the already-partially-anonymized text before it reaches the cloud. This catches names, company names, internal project references, and other context-dependent PII that regex misses. This second pass only runs on a local model — Argos enforces that sending raw content to a cloud LLM for anonymization would defeat the entire privacy model.

For local embeddings and vector search (knowledge base), Ollama with nomic-embed-text is also recommended but not required — you can skip the knowledge base entirely.

No, with one narrow exception. Every planned action — send a reply, create a calendar event, write a Notion page, send an email — is staged as a proposal that sits in your approval queue. Nothing executes until you explicitly approve it from the web dashboard or via the Telegram bot.

The one exception: actions in your own Notion workspace (creating personal tasks, notes) auto-execute without approval, because it's your private space and the risk is low. This can be disabled in config.

Proposals expire after 24 hours if not acted upon. There are no background cron jobs that fire outbound messages.

Argos supports 15+ providers out of the box: Anthropic (API key + OAuth), OpenAI, Google Gemini, Groq, DeepSeek, Mistral, xAI (Grok), Alibaba Qwen, Together AI, Perplexity, Cohere, Ollama, LM Studio, and any OpenAI-compatible endpoint.

You can configure a fallback provider — if your primary fails (rate limit, timeout, 5xx), Argos automatically retries on the fallback. And you can route different pipeline roles to different providers — e.g., use Ollama for sanitization/anonymization and Claude for planning.

The recommended setup is: a small local model (Ollama qwen2.5:7b or mistral:7b) for privacy-sensitive stages, and Claude Sonnet or GPT-4o for planning and drafting.

Telegram is optional. You can run Argos with only email, Slack, Discord, or WhatsApp if you prefer. Each channel is independently opt-in in the config.

That said, Telegram is the most deeply integrated channel — the bot mode lets you receive approval notifications, reply to proposals, trigger briefings, and run commands directly from your Telegram. Without it, you'll use the web dashboard at https://localhost:3000 for everything.

The Telegram MTProto listener (reads your personal account's messages) and the bot (handles approvals and commands) are separate — you can use one without the other.

Argos runs a local web server at https://localhost:3000 (TLS required). The dashboard shows your pending proposals, open tasks, recent memories, and system status.

Authentication uses WebAuthn (passkeys / YubiKey) — no passwords. A TOTP backup is available if you don't have a hardware key. High-risk proposals require a fresh WebAuthn assertion cryptographically bound to the specific proposal ID — you can't accidentally approve the wrong thing.

For on-the-go approvals, the Telegram bot mirrors the queue — you receive proposals as formatted messages and can approve or reject with a single tap, without opening the web app.

Argos itself is free and open source (MIT). The cost to run it depends entirely on your LLM usage.

At moderate message volume (a few hundred messages/day), API costs are typically a few dollars per month — most messages are short and many are filtered by the classifier before reaching the planner. The triage + classify pipeline is designed to be fast and cheap, deferring expensive planner calls to only when they're necessary.

With a local model for all privacy roles, cloud API costs drop significantly — only the planner calls (a fraction of total messages) hit the cloud. With a full local setup (Ollama for everything), there are no API costs at all, though response quality and tool-use reliability will be lower than with Claude or GPT-4o.

Yes. Argos has a TypeScript plugin API. A plugin is a plain object with lifecycle hooks: onBoot, onMessage, onShutdown. You register it with pluginRegistry.register() and it receives every message after the sanitize+anonymize pass.

You can also add custom tools to the planner by registering them in the tool catalog — the planner will use them when it decides they're relevant. MCP servers extend this further: add any MCP-compatible server to the mcpServers config and its tools are automatically available to the planner.

See src/plugins/examples/raw-forwarder.ts for a minimal plugin example, and src/plugins/README.md for the full API reference.

Node.js 22 or later. macOS, Linux, or Windows (WSL2 recommended on Windows). The process is lightweight — under 100MB RAM at rest for the core pipeline.

Optional but recommended: Ollama installed locally for the privacy LLM roles and vector embeddings. Ollama needs enough RAM to run your chosen model — 8GB for a 7B parameter model (qwen2.5:7b, mistral:7b), 4GB for a 3B model (llama3.2:3b).

The web dashboard requires a self-signed TLS certificate (instructions in the README) since WebAuthn only works over HTTPS, even on localhost.

Argos builds a personal memory store in SQLite with full-text search (FTS5). Memories are anonymized summaries extracted from processed conversations — never raw content. Each memory has a partner name, category, importance score, and a TTL (default 30 days).

When drafting a reply or planning an action, Argos searches this memory store for relevant context about the person or topic. High-importance memories (score ≥ 8) are auto-archived and kept for 365 days instead of expiring.

Separately, if you've configured the knowledge base (vector store), Argos also searches indexed documents (URLs, GitHub repos, Notion pages, local files) using semantic search before drafting. This is distinct from the conversational memory — it's your reference material, not past interactions.
